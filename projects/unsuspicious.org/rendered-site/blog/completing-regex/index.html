A quick overview of RegEx

Regular Expressions (RegEx) are a very common tool in computer science, often used for search-and-replace operations, finding specific strings or patterns in large files, and validating user input (see the infamous email regex).

In introductory classes to theoretical computer science, we see regular expressions as an example application of automata theory, more precisely finite automata. The process of converting a RegEx into an automaton is called Thompson’s construction, and is behind grep’s unreal speed.

The problem at hand

When building Proposition 7, I was struggling with a very peculiar problem in program synthesis and input validation.

Correctness by complete-ability

The p7 expression parser is based on a very simple principle:

With Σ\SigmaΣ an alphabet and LLL a language, we can define a function

Φ:Σ∗→{accept,reject}\Phi : \Sigma^* \rightarrow \{\text{accept},\text{reject}\}Φ:Σ∗→{accept,reject}

as follows:

Φ(s)={accept∃s′∈Σ∗, ss′∈Lrejectotherwise\Phi(s) = \begin{cases} \text{accept} & \exists s' \in \Sigma^*,\, ss' \in L \\ \text{reject} & \text{otherwise} \end{cases}Φ(s)={acceptreject​∃s′∈Σ∗,ss′∈Lotherwise​

Our goal here, for constrained generation, is to ensure that any prefix sss of a valid expression can be completed into a valid expression in LLL and thus lead to a valid program of proof.

RegEx sub-problem

Since our language LLL is defined by complex mechanisms, such as typing rules and recursive grammars, it must be handled by a Turing-complete model in the general case. However, a subset of this problem requires a generalized solution for RegEx.

Consider the following grammar as an example:

expr ::= /b(u|o|i)p/
term ::= expr ('and' expr)* 
prog ::= term 'at' 'bap'

expressions between slashes (/<regex>/) are regexes

The program bip at bap is a trivial valid expression, as is bip and bop at bap. However, bip and is not a valid expression, but can be completed into one by adding another expr + at bap after the and. We can call it Σ-completeable\Sigma\text{-completeable}Σ-completeable in LLL.

Now, consider that we are working with a system that produces output byte by byte. How can we validate the expression bi or bip an? Both are prefixes of completeable expressions, but neither is a Σ-completeable\Sigma\text{-completeable}Σ-completeable expression.

Formally, we could say that our grammar defines a language LLL over an alphabet Σ={/b(u|o|i)p/,’and’,’(’,’)’,’at’,’bap’}\Sigma = \{ \text{/b(u|o|i)p/}, \text{'and'}, \text{'('}, \text{')'}, \text{'at'}, \text{'bap'} \}Σ={/b(u|o|i)p/,’and’,’(’,’)’,’at’,’bap’} and that this alphabet is itself a language LΣL_\SigmaLΣ​ over the alphabet of bytes Σbyte=[0..255]\Sigma_{\text{byte}} = [0..255]Σbyte​=[0..255] (or more generally Σvocab\Sigma_\text{vocab}Σvocab​ corresponding to any vocabulary where Σvocab⊆Σbyte∗\Sigma_\text{vocab} \subseteq \Sigma_{\text{byte}}^*Σvocab​⊆Σbyte∗​ and Σ⊆Σvocab∗\Sigma \subseteq \Sigma_\text{vocab}^*Σ⊆Σvocab∗​).

This captures that we accept a partial input if either

  • It fully tokenizes into element of Σ\SigmaΣ and parses correctly in LLL
  • The partial token t′t't′ can be completed by some q∈Σvocab∗q \in \Sigma_{\text{vocab}}^*q∈Σvocab∗​ into a valid token l∈Σl \in \Sigmal∈Σ such that and appending lll to the complete tokens yields an expression that is Σ-completeable\Sigma\text{-completeable}Σ-completeable in LLL.

Keeping with our example, bi and bip an are both Σvocab-completeable\Sigma_{\text{vocab}}\text{-completeable}Σvocab​-completeable in LLL, as they can be completed into valid expressions in LΣL_\SigmaLΣ​: bip and bip and that are themselves Σ-completeable\Sigma\text{-completeable}Σ-completeable in LLL. On the other hand, bip at bo is not Σvocab-completeable\Sigma_{\text{vocab}}\text{-completeable}Σvocab​-completeable in LLL, because while it can be completed into the LΣL_\SigmaLΣ​ expression bip at bop, this expression is not Σ-completeable\Sigma\text{-completeable}Σ-completeable in LLL (it is even outright invalid).

Implementation

Focusing on the regex aspect of the program, we want to implement π(s)\pi(s)π(s) for the case where LΣL_\SigmaLΣ​ is defined by a regex rΣr_\SigmarΣ​. We cannot trivially do this by directly manipulating automata, but need a more general mechanism on the expressions themselves. In short, we need something that can tell us if a string s∈Σvocab∗s \in \Sigma_{\text{vocab}}^*s∈Σvocab∗​ is a valid prefix of a string in LΣL_\SigmaLΣ​.

One approach is find the set of suffixes S={s′∈Σvocab∗∣ss′∈LΣ}S = \{ s' \in \Sigma_{\text{vocab}}^* \mid ss' \in L_\Sigma \}S={s′∈Σvocab∗​∣ss′∈LΣ​}, and check if SSS is non-empty. The completing suffix of sss is known as the quotient of LΣL_\SigmaLΣ​ with respect to sss, and is written s−1LΣs^{-1}L_\Sigmas−1LΣ​. The expression that can produce this set of quotients is known as the Brzozowski derivative of LΣL_\SigmaLΣ​ with respect to sss, and is written Ds(LΣ)D_s(L_\Sigma)Ds​(LΣ​).

Correspondence to code

The Rust definition below implements DcD_cDc​ and then folds it over a string sss:

fn char_derivative(&self, c: char) -> Regex {
    use Regex::*;
    match self {
        Empty => Empty,
        Epsilon => Empty,
        Char(ch) => if *ch == c { Epsilon } else { Empty },
        Range(start, end) => if *start <= c && c <= *end { Epsilon } else { Empty },
        Union(a, b) => Union(Box::new((**a).char_derivative(c)), Box::new((**b).char_derivative(c))),
        Concat(a, b) => {
            let da = a.char_derivative(c);
            if a.nullable() {
                Union(
                    Box::new(Concat(Box::new(da), Box::new((**b).clone()))),
                    Box::new(b.char_derivative(c)),
                )
            } else {
                Concat(Box::new(da), Box::new((**b).clone()))
            }
        }
        Star(r) => Concat(Box::new((**r).char_derivative(c)), Box::new(Star(r.clone()))),
    }
}

pub fn derivative(&self, s: &str) -> Regex {
    s.chars().fold(self.clone(), |acc, c| acc.char_derivative(c))
}

Conclusion

In a real world implementaiton with grammar defined in EBNF-like format and an input sss over Σvocab\Sigma_\text{vocab}Σvocab​, we can implement Φ^(s)\hat{\Phi}(s)Φ^(s) as follows:

  1. Tokenize sss into tokens t1,t2,…,tnt_1, t_2, \ldots, t_nt1​,t2​,…,tn​ over Σ\SigmaΣ.
  • If tokenization succeeds and sss is fully tokenized, return Φ(s)\Phi(s)Φ(s)
  • If tokenization fails with remaning input t′t't′, continue to step 2
  1. Compute r′=Dt′(rΣ)r' = D_{t'}(r_\Sigma)r′=Dt′​(rΣ​) where rΣ=∑l∈Σrlr_\Sigma = \sum_{l \in \Sigma} r_lrΣ​=∑l∈Σ​rl​
  2. Build the set Tn+1={y∈Σ∣ ∃(tn+2,tn+3,...tn+k)∈Σ∗,Φ(t1t2...tn y tn+2,tn+3,...tn+k)=accept}T_{n+1} = \{y \in \Sigma | ~\exists (t_{n+2},t_{n+3},...t_{n+k}) \in \Sigma^*, \Phi(t_1t_2...t_n~y~t_{n+2},t_{n+3},...t_{n+k}) = \text{accept}\}Tn+1​={y∈Σ∣ ∃(tn+2​,tn+3​,...tn+k​)∈Σ∗,Φ(t1​t2​...tn​ y tn+2​,tn+3​,...tn+k​)=accept} of possible next continuations in LLL
  3. Check if ∃x∈Σvocab∗,l∈Σ\exists x \in \Sigma_\text{vocab}^*,l \in \Sigma∃x∈Σvocab∗​,l∈Σ such that ∃τ∈Tn+1,τ=l x\exists \tau \in T_{n+1}, \tau = l~x∃τ∈Tn+1​,τ=l x where ν(Dt′x(rl))=true\nu(D_{t'x}(r_l)) = \text{true}ν(Dt′x​(rl​))=true.

For the precise use case of constrained generation, Tn+1T_{n+1}Tn+1​ and r′r'r′ are passed to the sampling system and applied as filters on the model’s vocabulary Σvocab\Sigma_\text{vocab}Σvocab​ to only allow tokens that can complete sss into a valid expression.

Obviously, this is only a small piece in the puzzle, and a lot of challenges remain with the implementation of Φ\PhiΦ on arbitrary, context-dependent, languages, with context and type dependencies. It’s still cool to see how basic concepts like this have deep implications and practical value in real-world systems.
